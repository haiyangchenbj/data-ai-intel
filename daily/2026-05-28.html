1. Snowflake 用户实测：Adaptive Compute 在稳定 ETL 负载上反而劣于调优后的 Standard Warehouse

来源：Evaluating Adaptive Warehouses for ETL: Why We Reverted to Standard (dev.to)

核心观点：Snowflake 用户在 5/12-5/27 周期中将白银层 ETL Warehouse（约 2000 查询/日、负载稳定可预测）从 Standard 切换至预览版的 Adaptive Compute 模式，16 天后回退。实测数据：排队率从 ~0% 升至 3.4% 仍在攀升、P95 耗时从 14.1s 升至 20.4s、单查询 credits 从 0.0071 升至 0.0093（贵 31%）、平均排队时长从 3.3ms 升至 1543ms。作者结论：Adaptive Compute 设计假设是应对突发不可预测负载（dashboard、ad-hoc 分析），对节奏稳定的生产 ETL 管道反而是劣化方案，QUERY_THROUGHPUT_MULTIPLIER 锁在 3 倍这一限制成为关键瓶颈。

映射到数据平台的判断：这是一线工程团队对 Snowflake 推力旗舰算力新模型的少见公开打脸。说明两件事：（1）"AI 时代下一代弹性算力"的市场叙事和具体工作负载之间存在显著 gap，预览版本不应直接上生产 ETL；（2）数据平台的算力定价模型从"按 size 买"转向"按 query 买"是趋势，但在稳定批处理场景的成本表现仍未跑赢传统模式——结合 Snowflake 此次 $60 亿 AWS 承诺中"延展 Graviton 价格性能优势"的表述，平台方仍在向用户兑现传统按规格买算力的性价比承诺。CIO 在评估"Adaptive/Serverless"类新算力产品时需要按工作负载类型分别建模，不要一刀切迁移。
