趋势一：Agent-Native 数据平台成为全行业共识 ★★★（高可信）

核心判断：8 家厂商（Snowflake/Databricks/Google Cloud/腾讯云/火山引擎/阿里云/SelectDB/OceanBase）在 6 月以不同路径发布 Agent-Native 数据平台，跨中美两地、从独立平台到云厂商到开源社区，结构性证据充分。竞争维度已从"存储更多数据"转向"让 Agent 更高效地使用数据"。

补充观察：

Snowflake 将数据平台重新定义为 Agent 的"控制平面"——Openflow 开放多引擎治理、Adaptive Compute AI 自适应算力调度、CoWork 企业级 Agent 平台、Datastream 基于 Kafka 的托管流服务，四者指向"数据平台主动理解业务意图、自主执行数据任务"。Databricks 在 DAIS 2026 以 Genie One 连接 50+ 办公应用，Genie ZeroOps 实现管道故障自主检测修复；Unity AI Gateway 将治理从"谁能访问什么数据"推进到"Agent 在运行时可以做什么、花多少钱"。Google Cloud 以 Data Engineering Agent GA + Managed MCP Servers for Databases GA + MCP Toolbox 1.0 GA 快速补齐 Agent 化能力。

国内厂商呈现罕见的同步 Agent 化。腾讯云 DataBuddy 内测超 3000 家企业申请；WeData NL2SQL 准确率 90%+（腾讯云官方声称）。火山引擎数智平台将 Data Agent 从"数据专家"升级为"数字合伙人"。SelectDB 提出 CLI+Skill 范式——底层工具链重构为 CLI 接口。OceanBase LakeBase 统一数据湖开放存储、数据库事务/分析/多模态能力到底层引擎。

So what：数据平台负责人应将 Agent 治理能力（成本限额、行为审计、上下文追踪）作为 RFP 独立评估维度；采购方可利用当前多厂商同步发布窗口进行 POC，重点验证端到端延迟和成本可控性；竞品厂商若尚未包含 Agent 原生接口（MCP/CLI+Skill/自然语言驱动），建议在 Q3 前补齐。

趋势三：LTAP 架构终结 OLTP/OLAP 40 年分离范式 ★★（中可信）

核心判断：Databricks 在 DAIS Day 2 发布 LTAP（Lake Transactional/Analytical Processing）架构，首次在同一数据湖存储层上统一事务处理与分析。EDB Postgres AI 和 OceanBase 湖库一体从各自路线跟进。但大规模混合负载下的长期稳定性数据仍缺失，核心证据以 Databricks 单一厂商为主。

三条路线对比：

路线代表厂商核心设计关键声明（vendor-reported）
湖仓先行Databricks存储层统一：Postgres 事务引擎 + Spark 分析引擎，数据以 Delta/Iceberg 开放格式存储亚 100ms 延迟、12,000 QPS（Lakehouse//RT）；日均 1200 万次数据库启动（Lakebase）
运营层先行EDB关系型/分析型/向量/Agentic 工作负载统一到受治理的 Postgres 基础强调数据主权和可预测计费
湖库一体OceanBase数据湖开放存储 + 数据库事务/分析/多模态能力统一到底层引擎整体 TCO 降低 30%-50%（未经独立验证）

补充观察：

LTAP 不同于 HTAP 在引擎层强行融合（MemSQL、SAP HANA 等均未成功），而是在存储层统一，事务和分析工作负载各自独立扩展。Lakehouse//RT 基于全新 Reyden 计算引擎，直接在受治理的 Delta/Iceberg 表上运行。Lakebase 支持即时 Copy-on-Write 数据库分支——开发者可在秒级内创建生产数据库完整副本用于调试 AI Agent。EDB 以"运营层先行"路线挑战 Databricks；OceanBase 以"湖库一体"架构从中国市场切入。

预测性判断：若 LTAP 在 2026 年下半年获得至少 2 家头部企业核心系统的生产级验证，并公开独立基准测试数据，"是否需要独立事务数据库"将从默认答案变为可选项。当前仅基于 Databricks 单一厂商声明，置信度中等，建议持续跟踪至 2026 年 Q4 再做投资决策。

So what：数据平台负责人建议以 POC 为主，不建议在 2026 年下半年将核心交易系统迁移至 LTAP，除非有明确 SLA 承诺和独立基准验证；采购方应要求厂商提供与现有独立 OLTP + OLAP 架构的对比基准测试，重点关注混合负载下的尾延迟和故障恢复时间。

趋势四：数据治理从"数据权限"向"Agent 运行时管控"延伸 ★★（中可信）

核心判断：Agent 正在创造一种新的治理需求——不是静态的"谁能访问什么数据"，而是动态的"Agent 在运行时可以做什么、花多少钱、调用什么工具"。Databricks Unity AI Gateway 率先产品化，但行业标准尚未形成。核心证据以 Databricks 单厂商为主。

Immuta 在 Snowflake Summit 上发布 Agentic Data Access，为每个 Agent 交互分发唯一的临时角色，将治理粒度从"人能访问什么"推进到"Agent 在代表谁、能访问什么、每一步是否合规"。Databricks Unity AI Gateway 进一步将治理从数据访问扩展到 AI 运行时：覆盖模型、MCP 服务、Agent、技能的全生命周期，支持硬性支出上限、智能路由、上下文策略执行和统一追踪。Genie Ontology 为所有治理决策提供企业语义上下文——治理策略不再只是"允许/拒绝"，而是"基于业务语义理解允许什么"。

So what：数据平台负责人应将 Agent 治理能力作为独立评估维度，要求厂商提供运行时策略引擎的延迟指标和成本限额功能的精确度；采购方若采购周期紧迫，建议优先评估 Databricks Unity AI Gateway；若多平台战略是长期方向，建议等待 Snowflake 和 Informatica 后续跟进，避免单一厂商锁定。
